# Stria PDF Toolkit — Security Audit Report

**Date:** 2026-05-28  
**Auditor:** Internal (automated + manual review)  
**Version audited:** 0.2.0  
**Scope:** Full codebase — Rust/Tauri desktop shell, Python/FastAPI sidecar, TypeScript/React frontend  
**Standards mapping:** ISO 27001:2022 A.8.8 (technical vulnerability management), SOC 2 Type II CC7.1 (threat detection), CC6.1 (logical access)

---

## Executive Summary

Stria is a local-first PDF tool with no network connectivity to external services and no persistent document storage. This architecture fundamentally eliminates the most common web-app threat classes (server-side data breaches, session hijacking, CSRF). The attack surface is narrow: a local HTTP sidecar, a Tauri WebView, and a Rust command bridge.

**Before this audit:** 2 critical, 3 high, 4 medium findings.  
**After fixes applied:** 0 critical, 0 high, 1 medium (accepted risk), 3 low/info.

All critical and high findings have been remediated in this commit. The remaining medium finding (password transmission over local HTTP) is an accepted architectural risk documented below.

## Findings — Remediated

### [CRITICAL] PATH-TRAVERSAL — Arbitrary file read via `read_file_bytes`

**Severity:** Critical  
**Component:** `src-tauri/src/lib.rs` — `read_file_bytes` Tauri command  
**ISO 27001:** A.8.8, A.8.28  
**SOC 2:** CC6.1  

**Description:**  
The original `read_file_bytes` command accepted any filesystem path from the WebView JS layer with no validation:

```rust
// VULNERABLE (original)
fn read_file_bytes(path: String) -> Result<Vec<u8>, String> {
    std::fs::read(&path).map_err(|e| format!("Failed to read {}: {}", path, e))
}
```

If malicious content loaded in the WebView (e.g. via a crafted PDF's embedded JavaScript, or a future XSS) could invoke Tauri commands, it could read any file the user has access to: SSH keys, credential stores, `%APPDATA%\*`, etc.

**Fix applied:**  
Introduced `validate_pdf_path()` which:
1. Calls `Path::canonicalize()` — resolves `..` traversal sequences and symlinks, and verifies the file exists
2. Checks the canonical path's extension is `.pdf` (case-insensitive)
3. Returns a deliberate vague error message (no path leakage)

The fix is applied consistently in `read_file_bytes`, `get_cli_file_path`, and the single-instance `open-file` event emitter.

---

### [CRITICAL] PATH-TRAVERSAL — Same issue in single-instance handler

**Severity:** Critical (same root cause)  
**Component:** `src-tauri/src/lib.rs` — `single_instance` plugin handler  

**Description:** The `open-file` event emitted to the frontend when a second Stria instance is launched was passing the raw CLI argument path without validation. A crafted file association (e.g. `stria.exe ../../secret.pdf`) could cause the frontend to read an unintended file.

**Fix applied:** The same `validate_pdf_path()` call wraps the path before it is emitted.

### [HIGH] MISSING-CSP — Content Security Policy disabled

**Severity:** High  
**Component:** `src-tauri/tauri.conf.json`  
**ISO 27001:** A.8.28  

**Description:**  
`"csp": null` disables all WebView content security protections. A WebView with no CSP cannot prevent injected scripts from making cross-origin requests, loading remote resources, or calling `eval()`.

**Fix applied:**  
```json
"csp": "default-src 'self'; script-src 'self' 'unsafe-inline'; style-src 'self' 'unsafe-inline'; img-src 'self' blob: data:; connect-src 'self' http://127.0.0.1:7342; font-src 'self' data:; worker-src blob:; object-src 'none'; base-uri 'self'"
```

Notes:
- `'unsafe-inline'` for scripts is required by PDF.js's worker bootstrapping and React's development mode. For a production build the inline scripts should be replaced with nonces or hashes in a future hardening pass.
- `connect-src` is restricted to `self` + the sidecar's exact address; external network calls are blocked.
- `object-src 'none'` prevents Flash/plugin execution.

### [HIGH] OVERLY-PERMISSIVE-CORS — `allow_headers: ["*"]` and dev origin in production

**Severity:** High  
**Component:** `backend/main.py`  

**Description:**  
`allow_headers=["*"]` permitted any request header, including non-standard headers that could be used to smuggle data. While the sidecar only binds to 127.0.0.1 and the CORS list only includes localhost origins (not `*`), the wildcard header allowance was unnecessary.

**Fix applied:**  
```python
allow_methods=["GET", "POST"],
allow_headers=["Content-Type", "Accept"],
expose_headers=["Content-Disposition"],
```

---

### [HIGH] MISSING-SEC-HEADERS — No security response headers from sidecar

**Severity:** High  
**Component:** `backend/main.py`  
**ISO 27001:** A.8.28  

**Description:**  
The FastAPI sidecar returned responses with no security headers, allowing content-sniffing, clickjacking via iframes, and caching of sensitive PDF bytes.

**Fix applied:** Added HTTP middleware that appends to every response:
- `X-Content-Type-Options: nosniff`
- `X-Frame-Options: DENY`
- `Cache-Control: no-store` (prevents browser/proxy caching of PDF content)
- `Referrer-Policy: no-referrer`

---

### [HIGH] INPUT-VALIDATION — Watermark text unbounded length (DoS)

**Severity:** High  
**Component:** `backend/routers/watermark.py`  

**Description:**  
The watermark `text` form field had no maximum length. A 100 MB text string would cause PyMuPDF's `TextWriter` to allocate memory proportional to the string length times number of pages, potentially causing OOM or extreme latency.

**Fix applied:** `len(text) > 200` → HTTP 400.

---

### [HIGH] INPUT-VALIDATION — Annotation fields unbounded (DoS + type confusion)

**Severity:** High  
**Component:** `backend/routers/annotate.py`  

**Description:**  
- No max length on `text`, `label` fields → unbounded string processing by PyMuPDF
- No bounds check on coordinate values (`x0`, `y0`, `x1`, `y1`) → `NaN`/`Infinity` values could cause undefined PyMuPDF behavior
- No limit on number of annotations, strokes, or stroke points → DoS via oversized payloads
- No type check on coordinate fields → passing a string where a float is expected could cause unexpected Python coercion

**Fix applied:**  
Introduced `_is_unit_float()` helper (checks `isinstance`, `isnan`, `isinf`, `[0.0, 1.0]` range). Applied to all coordinate fields. Added caps:
- `MAX_TEXT_LEN = 10_000` per annotation text field
- `MAX_LABEL_LEN = 200` for stamp labels
- `MAX_STROKES = 500` per ink annotation
- `MAX_POINTS = 2_000` per stroke
- `MAX_ANNOTS = 2_000` per save operation

---

### [HIGH] INPUT-VALIDATION — Redact coordinates not type/range validated

**Severity:** High (same class as annotation findings)  
**Component:** `backend/routers/redact.py`  

**Description:**  
The original validation only checked key presence, not types or value ranges. `{"page": 1, "x0": "NaN", "y0": Infinity, "x1": 0, "y1": 0}` would pass validation and be passed directly to PyMuPDF.

**Fix applied:** Same `_is_unit_float()` pattern, page number bounds check, `MAX_REGIONS = 2_000` cap.

---

### [HIGH] INFO-LEAK — Docs/Redoc/OpenAPI exposed in production

**Severity:** High (information disclosure)  
**Component:** `backend/main.py`  

**Description:**  
FastAPI enables interactive API documentation at `/docs`, `/redoc`, and `/openapi.json` by default. In the bundled sidecar, this exposes the full API schema to any process that can reach port 7342.

**Fix applied:**  
```python
docs_url=None if getattr(sys, "frozen", False) else "/docs",
redoc_url=None if getattr(sys, "frozen", False) else "/redoc",
openapi_url=None if getattr(sys, "frozen", False) else "/openapi.json",
```
Docs remain available in development but are disabled in the production bundle.

## Findings — Accepted Risk

### [MEDIUM] LOCAL-CLEARTEXT — Passwords sent over local HTTP

**Severity:** Medium (accepted)  
**Component:** `backend/routers/security.py`, `src/api/client.ts`  
**ISO 27001:** A.8.24  

**Description:**  
The `encryptPDF` and `decryptPDF` functions transmit the user's password in plaintext over `http://127.0.0.1:7342`. A process running as the same user could intercept this via loopback sniffing or by reading process memory.

**Risk acceptance rationale:**  
- The sidecar binds exclusively to `127.0.0.1` (not `0.0.0.0`), so interception requires local code execution as the same user — at which point the attacker already has full access to the user's files anyway.
- The alternative (implementing encryption in a Tauri command using a Rust binding to pikepdf) would require significant architectural work and introduces a different attack surface.
- This is consistent with how password-manager local agents, developer tools, and browser password stores handle local IPC.

**Mitigation (future):** Move `encrypt`/`decrypt` to a Tauri command using a Rust crypto binding, eliminating the HTTP transport entirely.

## Findings — Informational

### [LOW] SIDECAR-NO-AUTH — Sidecar port is unauthenticated

**Description:**  
Port 7342 has no authentication token. Any process running as any user on the machine can call the PDF processing API. This is an intentional design choice for a local tool (simplicity > auth overhead for localhost), but should be documented.

**Mitigation (future):** Generate a random token at sidecar startup, pass it to the frontend via Tauri's `emit`, and require it as an `Authorization: Bearer` header on all API calls.

---

### [LOW] DEPS-UNPINNED — npm dependencies use `^` semver ranges

**Description:**  
All `package.json` dependencies use `^` ranges. `npm ci` with a committed `package-lock.json` mitigates this in practice, but the lock file must never be deleted or regenerated without review.

**Recommendation:** Add a CI step that fails if `package-lock.json` was not committed, or migrate to exact pinning (`--save-exact`).

---

### [LOW] SIDECAR-UNSIGNED — PyInstaller binary is not code-signed

**Description:**  
The bundled `pdftools-server.exe` sidecar is not signed with a code-signing certificate. Windows SmartScreen will warn users on first run.

**Recommendation:** Obtain an EV code-signing certificate for distribution builds.
